= ONTAP 外部 LUN 导入 (FLI) 分析阶段
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
分析阶段确定在进入详细的迁移规划和执行之前必须修复的内容。此阶段的主要结果是确保每个主机及其所依赖的完整 SAN 堆栈（操作系统、卷管理器、HBA、多路径、交换机固件和目标 ONTAP 版本）可以被纳入支持的迁移后配置。这是通过将每个主机的当前状态与 NetApp 互操作性矩阵（IMT）中支持的组合进行比较，然后为每个主机定义明确的目标配置（操作系统级别、MPIO/ALUA 行为、HBA 驱动程序/固件和任何所需的主机工具），并生成差距分析，列出达到该目标所需的确切更改来完成的。

安排和审查差距分析，以避免补救更改，这可能会影响应用程序兼容性和中断计划。在许多环境中，某些更改（如 OS 修补或 HBA 更新）在迁移窗口之前完成更安全。但是，多路径堆栈更改通常会延迟到切换或迁移后，因为更改 MPIO 行为可能会影响当前阵列的可支持性并使回滚复杂化。

对于 iSCSI FLI，分析阶段还应捕获 iSCSI 特定的准备情况：

* 确认主机的 iSCSI 多路径设计在切换后仍受支持
* 确保外部阵列符合 iSCSI 后端使用条件，因为后端传输差异可能会改变互操作性行为。




== ONTAP FLI 分析阶段任务

分析阶段以主机为中心，但也应验证完整的端到端支持链（主机到交换矩阵再到 ONTAP 然后返回）并确认 FLI 后端互操作性状况。

[cols="1,2"]
|===
| 组件 | 任务 


| 主机  a| 
* 将当前主机配置与 IMT 支持的配置进行比较，并确定目标迁移后配置：
+
** OS 级别
** 多路径方法
** HBA 型号、驱动程序和固件
** 所需的主机工具


* 创建每个主机的差距分析，并提供所需的：
+
** 热修复和补丁
** 操作系统更新
** HBA 驱动程序和固件升级
** 多路径更改
** 列出任何所需的实用程序






| 结构 (FC/iSCSI 网络)  a| 
* 在端到端视图中验证交换机和结构支持（型号和固件），如果目标 ONTAP 和主机组合不支持该结构，则记录所需的升级。




| NetApp 存储阵列（目标）  a| 
* 确认目标 ONTAP 平台和版本与预期的 FLI 工作流程兼容，包括任何适用的 AFF 启用预期，并记录影响目标设计的任何先决条件。




| 外部存储阵列（源）  a| 
* 验证源阵列是否受 FLI 后端互操作性支持。如果 IMT 中未列出 ONTAP 9.9.1 及更高版本的互操作性，请使用 SAN LUN Migrate App 来帮助确定是否可能受支持。


|===


== 在 ONTAP FLI 分析阶段使用 IMT

互操作性矩阵工具 (IMT) 是一个配置存储库，也是一种缩小到受支持的端到端结果配置的引导方式，其中 NetApp 产品与第三方组件互操作。IMT 可以包含受支持和认证的配置：

* 支持的配置由 NetApp 认证。
* 经认证的配置由第三方认证，可与 NetApp 组件配合使用。


.检查 IMT 以了解：
* 在工作表中捕获 IMT 操作：在规划工作表中记录 IMT 推荐的主机和交换机软件、修补程序和升级。
* 从广泛开始，然后细化：从静态标准（ONTAP 版本、协议、HA/群集模式）开始，然后使用站点调查作为指南添加主机操作系统、卷管理器和 HBA 详细信息。
* 避免过度筛选：如果您的搜索过于具体，没有返回任何结果，请放松筛选条件，查看多个有效配置，并选择最受支持的匹配项。
* 规范化 HBA 标识符：HBA 可能显示为 OEM 零件号，在将其输入 IMT 之前请交叉引用它们。
* 检查每台主机：对迁移范围内的每台主机运行 IMT 支持性检查。
* 执行两次互操作性检查：
+
** 后端 FLI 支持：确认支持源阵列后端场景。
** 端到端迁移后支持：确认支持 ONTAP、主机操作系统、多路径、HBA 和交换矩阵组合。


* 实用 IMT 工作流：
+
** 搜索源阵列模型，然后选择"Foreign LUN Import (FLI) backend Interoperability"，然后选择 ONTAP 平台和 ONTAP 版本。
** 对于主机兼容性，请使用"Build end-to-end view with ONTAP SAN host"来验证操作系统、多路径和 HBA。
** 要获得交换机兼容性，请从 SAN 主机视图中使用"为 SAN 交换机构建端到端视图"来验证结构和交换机支持。




.如果外部阵列未在 IMT 中列出（ONTAP 9.9.1 及更高版本）
从 ONTAP 9.9.1 开始，如果外部阵列未在 IMT 中列为受支持，则可以使用 SAN LUN 迁移工具来帮助确定阵列是否可能支持 FLI。该应用程序从 NetApp 支持站点下载，并从具有块访问权限（FC 或 iSCSI）的 Linux 主机运行到源阵列 LUN。结果将与工程人员一起审查。从外部阵列导入可以是现场合格的。对于 iSCSI FLI，从外部阵列导入始终是现场合格的。



== ONTAP FLI 主机更改和回滚策略

分析阶段应明确决定更改的内容和时间，因为主机更改可能会带来风险，也会影响对当前存储的支持。如果主机堆栈在能够获得 ONTAP 支持之前需要升级，这一点很重要。

* 在可能的情况下提前完成主机更新（修补程序、HBA 固件和驱动程序）可以更安全，但仍需要与维护窗口和应用程序团队进行协调。
* 对多路径堆栈（MPIO、ALUA 和第三方多路径）的更改可能会影响当前阵列的可支持性，因此它们通常会推迟到迁移事件甚至切换之后。
* 如果必须在中断窗口期间还原到原始存储，则延迟 MPIO 重新配置可以使回滚更简单。




== ONTAP FLI 间隙分析

差距分析总结了当前环境与 NetApp 推荐的目标配置，并列出了实现支持的迁移后状态所需的升级或补救措施。

每个主机的完整目标配置通常包括：

* 操作系统配置级别和所需的修补程序或热修复程序
* 多路径方法和所需设置（MPIO 和 ALUA 行为
* HBA 和 CNA 型号、驱动程序和固件版本
* 与您的环境相关的主机实用程序和工具要求（如适用）


应及时审查差距分析，因为补救措施可能需要停机时间窗口，并且可能与应用程序支持要求相互作用。
